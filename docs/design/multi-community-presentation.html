Multi-Community Support

From a single-identity tool to joining and operating in many Verifiable Trust Communities at once — split setup, per-community identities, a Communities hub, and concurrent live sessions.

Spec: docs/design/multi-community-support.md · DRAFT v4 Plan: tasks/plan.md · 9 tasks Decisions D1–D17

The shift in one slide

What changes

Today
  • Profile-singleton: one persona DID, one VTA, one mediator, one flat context.
  • One monolithic ~19-step setup that creates everything up front.
  • A single DIDComm session / message loop.
  • No notion of communities, joining, or membership.
After
  • Account → many personas → many communities.
  • Setup splits: State A bootstrap (no DID) + State B join.
  • Concurrent live sessions — one supervised task per community.
  • A Communities hub; the main page is community-scoped with a switcher.
  • VTA is the store of keys/credentials; local config is metadata.

Deferred on purpose: Verifiable Presentation construction (join submits a stub VP) and the VTA-side sub-context authorization model.

Starting point

Today: one identity, one authority

Each profile is a closed singleton. Useful, but it can't represent belonging to more than one community.

Flow · Membership lifecycle

Community states

Only Active is live; every other state is read-only (D14) and can be archived or deleted.

stateDiagram-v2 [*] --> Pending: submit join Pending --> Active: accepted Pending --> Rejected: denied Pending --> Expired: 7 days, no reply Active --> Left: self-remove Active --> Removed: removed by VTC Left --> Pending: re-join Rejected --> Pending: re-join Expired --> Pending: re-join Removed --> Pending: re-join Rejected --> [*]: delete Left --> [*]: delete Removed --> [*]: delete Expired --> [*]: delete

Rejected, Removed, Expired, more-info raise the actions-required badge until acknowledged (R-S-2).

Before / After · Runtime

Single loop → supervised multi-session manager

Today

After · D11/D15

One supervised task per active community: independently recoverable, bounded concurrency, partial-failure-tolerant launch. A mediator outage on one community never blocks the others.

Decisions of record

D1–D17 at a glance

#Decision
D1Per-join: mint a new persona or reuse an existing one (user chooses)
D5/D6No DID at bootstrap; personas are self-contained & context-independent
D2/D9Hierarchy via <top>/<slug> convention behind one context_path module
D7Mediator defaults to the VTA's; optional per-DID override
D8/D14States: Pending · Active · Left · Rejected · Removed · Expired — non-Active = read-only (archive/delete)
D10/D13Full refactor, no shim; old config = breaking reset (warn + delete), not migration
D11/D15Concurrent live sessions; one supervised, isolated task per community
D12VTA is the store of persona DIDs / keys / credentials
D16Persona+session before submit; 7-day pending timeout; VTC may reject / request more info
D17did-git-sign selects the signing persona via env var / per-repo git config

Status & what's next

Deferred, open, and the ask

Deliberately deferred

  • Verifiable Presentation construction (join uses a stub VP)
  • Real hierarchical context API (parent_id)
  • New per-community capabilities beyond porting today's main page
  • Persona key rotation

Open — external dependencies

  • VP requirement discovery — how a VTC advertises required credentials
  • Sub-context authorization — to be designed at the VTA layer

The ask

Sign off on D1–D17 & the plan, then start T1 (with an active-identity API sketch for review first).
